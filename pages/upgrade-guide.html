Migration

Upgrade guide

JSZipp can replace JSZip in many browser workflows, but it is not an API-compatible drop-in.

Important: JSZipp is not a drop-in replacement for JSZip. JSZip gives you a mutable archive object; JSZipp gives you append-style writers and readers.

Migration dealbreakers

If your code depends on any of these JSZip patterns, the migration is a reshape, not a method rename.

1. Fluent folder and file chaining

JSZip supports a mutable folder tree. JSZipp does not expose nested folder objects; pass full ZIP paths instead.

// JSZip
const zip = new JSZip();
zip.folder("images")?.file("logo.png", data);

// JSZipp
import { ZipWriter } from "web-jszipp";

const writer = new ZipWriter({ outputAs: "blob" });
await writer.add({ path: "images/logo.png", data });
const zipBlob = await writer.close();

Directories are explicit entries when you need them:

await writer.add({ path: "images/", data: "" });

2. In-place modifications

JSZip lets you mutate a loaded archive. JSZipp does not. To remove or replace files, rebuild a new archive from the old one.

import { ZipWriter, openZip } from "web-jszipp";

const source = await openZip(input);
const writer = new ZipWriter({ outputAs: "blob" });

for (const entry of source.entries) {
  if (entry.path === "temp.txt") continue;
  await writer.add({
    path: entry.path,
    data: entry.isDirectory ? "" : await entry.bytes(),
    meta: {
      comment: entry.comment,
      extraField: entry.extraField,
      modifiedAt: entry.modifiedAt,
      externalAttributes: entry.externalAttributes
    }
  });
}

await writer.add({ path: "report.txt", data: newReport });
const output = await writer.close();
await source.close();

That pattern rebuilds the archive and materializes each copied entry when entry.bytes() is used.

3. Synchronous input limits

writeSync() and closeSync() exist, but only for synchronously readable data.

const writer = new ZipWriter({ outputAs: "uint8array" });
writer.writeSync({ path: "hello.txt", data: "Hello" });
const bytes = writer.closeSync();

writeSync() accepts string, Uint8Array, and ArrayBuffer. Use async add() for Blob and ReadableStream<Uint8Array>. Do not mix async and sync modes on one writer.

4. Legacy browser and Node-specific APIs

JSZipp targets modern browser primitives and does not directly provide JSZip's Base64 or Node Buffer output modes. If you need older browsers or Node-oriented ZIP outputs from the library itself, staying on JSZip may be the simpler choice.

Create a ZIP Blob

import { ZipWriter } from "web-jszipp";

const writer = new ZipWriter({ outputAs: "blob" });
await writer.add({ path: "images/logo.png", data });
const blob = await writer.close();

Read one file

import { openZip } from "web-jszipp";

const zip = await openZip(file, { pathMode: "strict-package" });
const readme = await zip.get("README.md")?.text();
await zip.close();

Where JSZipp fits better

1. Smaller browser-focused dependency surface

JSZipp has no runtime dependencies. ZIP writing uses its in-repo raw DEFLATE encoder; reading deflated entries delegates inflation to the platform's DecompressionStream when needed.

2. Web-native output shapes

ZipWriter can return a ReadableStream, Blob, Response, Uint8Array, or ArrayBuffer.

const stream = await new ZipWriter().close();
const blob = await new ZipWriter({ outputAs: "blob" }).close();
const response = await new ZipWriter({ outputAs: "response" }).close();

The default output is a ReadableStream<Uint8Array>. That is useful for browser pipelines and service-worker-style responses, but it is still not true streaming compression of each source file.

3. ZIP64 writing and reading

JSZipp supports ZIP64 in both writer and reader code. Use zip64: "off" if you want standard ZIP only and prefer a hard error when ZIP64 would be required.

const writer = new ZipWriter({
  zip64: "auto",
  outputAs: "blob"
});

4. Safer path defaults when reading

openZip() rejects unsafe paths by default, including .., absolute paths, drive-letter paths, backslash-separated paths, and NUL-byte paths.

const reader = await openZip(file, { pathMode: "strict" });

For untrusted packages or uploads, use pathMode: "strict-package" to add a local/central size cross-check and reject duplicate or colliding entry paths after case and Unicode normalization.

const reader = await openZip(untrustedUpload, {
  pathMode: "strict-package"
});

Common JSZip to JSZipp translations

Create a ZIP Blob

// JSZip
const zip = new JSZip();
zip.file("hello.txt", "Hello");
const blob = await zip.generateAsync({ type: "blob" });

// JSZipp
const writer = new ZipWriter({ outputAs: "blob" });
await writer.add({ path: "hello.txt", data: "Hello" });
const blob2 = await writer.close();

Read one file

// JSZip
const zip = await JSZip.loadAsync(file);
const text = await zip.file("docs/readme.md")?.async("text");

// JSZipp
const reader = await openZip(file);
const text2 = await reader.get("docs/readme.md")?.text();
await reader.close();

Preserve duplicate names

JSZipp exposes archive order directly. The default reader preserves duplicate paths; reader.get(path) returns the most recently appended matching entry.

const reader = await openZip(file);
const copies = reader.entries.filter((entry) => entry.path === "data.json");
const latest = reader.get("data.json");

Decision matrix

Stay on JSZip if

  • Your code depends on zip.folder().file() and a mutable archive object.
  • You need to remove or overwrite files inside a loaded archive before generating output.
  • You need Base64 or Node Buffer output modes from the ZIP library itself.
  • You target older browsers or broad legacy environments.

Consider JSZipp if

  • Your app targets modern browsers and Web Streams.
  • You want a compact, dependency-free ZIP package.
  • Your output naturally wants ReadableStream, Blob, Response, Uint8Array, or ArrayBuffer.
  • You need ZIP64 writing or reading within JavaScript's safe integer range.
  • You can rebuild archives instead of mutating them in place.
Do not migrate solely for one huge file: JSZipp's public APIs are stream-shaped, but the current writer still materializes each entry before emitting it.
